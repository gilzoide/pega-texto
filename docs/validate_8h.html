Possible status codes for Grammar validation.

Enumerator
PT_VALIDATE_SUCCESS 

No errors on validation.

PT_VALIDATE_NULL_GRAMMAR 

Grammar is a NULL pointer.

PT_VALIDATE_EMPTY_GRAMMAR 

Grammar doesn't present any Rules.

PT_VALIDATE_NULL_POINTER 

NULL pointer as Expression data.

PT_VALIDATE_RANGE_BUFFER 

Range buffer must have at least 2 characters.

PT_VALIDATE_INVALID_RANGE 

Range characters must be numerically ordered.

PT_VALIDATE_OUT_OF_BOUNDS 

Non-terminal index is out of Grammar bounds.

PT_VALIDATE_UNDEFINED_RULE 

Rule undefined in given Grammar.

PT_VALIDATE_LOOP_EMPTY_STRING

◆ pt_validate_behaviour

enum pt_validate_behaviour

Behaviours for Grammar validation.

Enumerator
PT_VALIDATE_DEFAULT 

Default behaviour: validate when asked, without printing errors nor aborting.

PT_VALIDATE_SKIP 

Skip well-formedness validation.

This still looks for NULL pointers, as they may be introduced by malloc errors.

This is useful for release builds when using non-changing Grammars you know are valid.

PT_VALIDATE_PRINT_ERROR 

Print an error message on stderr if Grammar is invalid.

PT_VALIDATE_ABORT 

Abort program if Grammar is invalid, printing an error message on stderr.

May be useful for Debug builds.

Validate a Grammar.

This checks if a Grammar is well-formed, as described by Ford (2014), and if Non-terminal indexes (either by name or numerical index) exist and are inbounds.

Parameters
gGrammar to be validated.
bhvValidation behaviour
Returns
Validate result: a {status code, invalid rule index (if there is one)} pair.
Examples:
InfixCalculator.c, Lisp.c, and Re.c.
